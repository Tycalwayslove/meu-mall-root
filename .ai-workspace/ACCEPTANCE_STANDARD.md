# 统一验收标准

## 验收维度

每个正式工作项必须从四个维度验收。

## 产品验收

确认用户路径成立：

- 成功状态可用。
- loading 状态清楚。
- error 状态可恢复。
- empty 状态有明确反馈。
- 权限或登录状态有预期处理。

## 工程验收

确认实现符合架构边界：

- 没有绕过共享 API client、Bridge adapter、manifest runtime 等边界。
- 没有把其他项目职责写进当前项目。
- 没有引入未说明的新依赖。
- 错误处理不会导致整页白屏或服务崩溃。
- 类型、schema 或数据结构变更有兼容说明。

## 测试验收

确认验证有记录：

- 单元测试覆盖核心分支。
- 契约测试覆盖跨项目接口。
- 构建或类型检查通过。
- 无法自动化验证时，写明人工验证步骤和结果。

## 发布验收

确认上线可控：

- 明确是否影响 manifest。
- 明确是否需要灰度。
- 明确回滚目标。
- 明确 smoke check。
- 明确上线后观察指标。

## 最低验收口径

一个任务不能只以“代码写完”作为完成标准。至少要有：

```text
实现结果
验证命令
验证结果
风险说明
状态更新
```
