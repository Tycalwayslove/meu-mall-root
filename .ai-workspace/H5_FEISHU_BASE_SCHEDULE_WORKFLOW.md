## 目标

本工作流用于把 MeuMall H5 页面开发、后端接口、原生 App 对接和测试验收统一放进飞书多维表格中管理。

它解决的问题是：

- 页面很多，但不知道每个页面依赖哪些接口、哪些原生能力、哪些测试场景。
- 后端、原生、测试各自补充信息时没有统一入口。
- 需求排期只写“大阶段”，无法追踪单个页面是否真正可联调、可测试、可验收。
- 新开 AI 会话或新成员接手时，无法快速理解当前页面、接口、Bridge 和测试状态。

## 核心原则

- 页面是主需求。
- 后端接口是页面的子需求。
- 原生 App 对接是页面的子需求。
- 测试验收是页面的子需求。
- 版本迭代只负责收拢页面，不替代页面级状态。
- Base 用于协作和排期，仓库文档仍是长期事实源。

### 表关系

```text
版本迭代排期 1 -> N H5 页面需求
H5 页面需求 1 -> N 后端接口需求
H5 页面需求 1 -> N 原生 App 对接需求
H5 页面需求 1 -> N 测试验收需求
```

每张子表都必须关联回 `H5 页面需求`。禁止只在备注里写“需要后端接口”或“需要 App 配合”。

### 显示名和编号

Base 关联展示优先使用主显示字段。为了让关联字段对人友好，同时保留稳定编号，所有需求表采用“双字段”：

| 表 | 主显示字段 | 稳定编号字段 |
| --- | --- | --- |
| H5 页面需求 | 页面显示名称，例如 `推广首页｜/promotion` | 页面需求 ID |
| 后端接口需求 | 接口显示名称，例如 `推广首页数据｜H5 BFF` | 接口需求 ID |
| 原生 App 对接需求 | 原生显示名称，例如 `打开搜索 WebView｜L-006` | 原生需求 ID |
| 测试验收需求 | 测试显示名称，例如 `权益中心等级切换｜功能` | 测试需求 ID |

文档、契约和自动化仍然使用稳定编号；飞书表格中的关联展示使用中文显示名。

### 责任组和人员占位

每张需求表都必须有：

- `责任组`：`前端`、`后端`、`原生`、`测试`、`iOS`、`Android`、`产品`、`设计`、`项目管理`、`待定`。
- `协作成员`：关联 `团队人员与角色`。

在真实人员未确定前，使用占位成员：

| 占位成员 | 用途 |

| 后端待分配 | Java、Python、BFF、配置接口任务。 |

原生对接需求的总责任组统一写 `原生`；iOS 和 Android 的具体状态分别看 `iOS 状态`、`Android 状态`，协作成员同时关联 `iOS待分配` 和 `Android待分配`。

### 协作总览

为了让前端、后端、原生和测试不用在多张表之间来回切换，Base 中必须保留 `项目协作总览` 表。

这张表是统一任务索引，面向日常协作，第一屏字段必须按以下顺序优先展示：

1. 协作任务
2. 任务类型
3. 当前状态
4. 优先级
5. 责任组
6. 协作成员
7. 计划开始
8. 计划完成
9. 缺啥/阻塞项
10. 需要谁提供
11. 下一步动作

使用原则：

- 开发看 `全员协作总览`、`开发个人任务`、`按责任组看任务`。
- 项目负责人看 `阻塞和待补充`、`协作任务甘特图`。
- 领导看 `领导进度总览` 和 `MeuMall H5 进度仪表盘`。
- 页面、接口、原生、测试的细节仍回到各子表维护。
- `项目协作总览` 是索引层，后续新增需求时需要由 AI 或项目负责人同步新增对应任务行。

同步口径：

- `来源需求 ID` 是跨表同步键，必须和源表稳定编号一致。
- `来源表` 必须填写真实源表名，例如 `H5 页面需求`、`后端接口需求`。
- 原生对接表有 `iOS 状态` 和 `Android 状态` 两个字段；从总览修改原生任务时会同时更新双端状态，从任一端明细状态修改时，最后一次修改会回写总览 `当前状态`。
- `项目协作总览.当前状态` 已补齐版本状态选项 `可提测`、`验收中`、`已发布`，但当前总览中没有版本类型记录，因此暂未启用版本状态同步 workflow。
- 自动化同步有短暂延迟，通常不是即时同屏刷新；刷新页面或等待数秒后可见。

## 状态口径

### 页面开发状态

| 状态 | 说明 |
| --- | --- |
| 未开始 | 页面尚未进入设计确认或开发。 |
| 需求澄清 | 页面范围、路由、端归属或验收还没确认。 |
| 设计确认 | Figma 和页面元素已确认，可以拆任务。 |
| 静态页面 | H5 已完成静态或 mock 页面，但未接真实接口或真实 Bridge。 |
| 开发中 | H5 正在实现正式页面能力。 |
| 待联调 | H5 侧已具备联调条件，等待后端或 App。 |
| 联调中 | 正在和后端、App、测试环境联调。 |
| 可测试 | 已满足测试进入条件，QA 可以执行用例。 |
| 可验收 | 核心用例通过，产品或负责人可以验收。 |
| 已完成 | 验收通过且发布影响已记录。 |
| 阻塞 | 存在 P0 阻塞，无法继续推进。 |
| 暂缓 | 本阶段不排期。 |

### 子需求状态

后端、原生和测试统一使用以下状态，减少跨团队理解成本：

| 状态 | 说明 |
| --- | --- |
| 待补充 | 需要对应团队补充细节。 |
| 待确认 | 已有草案，等待负责人确认。 |
| 已确认 | 范围和交付口径明确。 |
| 开发中 | 对应团队正在实现。 |
| 自测完成 | 对应团队已完成自测。 |
| 联调中 | 正在跨团队联调。 |
| 可测试 | 已交付给测试。 |
| 通过 | 该子需求验收通过。 |
| 阻塞 | 无法继续，必须写明阻塞原因。 |
| 暂不需要 | 当前页面明确不需要此类配合。 |

## 页面进入可测试的门槛

页面不能只因为 H5 写完就标记为 `可测试`。至少满足：

- H5 静态资源、路由、loading、empty、error、未登录或 token 过期状态可用。
- 关联的 P0 后端接口状态为 `可测试`、`通过`，或明确当前阶段使用 H5 mock。
- 关联的 P0 原生 App 对接状态为 `可测试`、`通过`，或明确当前页面不需要原生能力。
- 至少存在一条测试验收需求，且测试用例状态不是 `待补充`。
- 当前阻塞项为空，或只剩 P1/P2 非阻塞事项。

## 页面进入可验收的门槛

页面进入 `可验收` 前，必须满足：

- 测试验收需求中的 P0 用例全部通过。
- 后端接口契约和实际响应一致，或 mock 阶段风险已标注。
- 原生 App Bridge、WebView 容器、返回、切 Tab 等对接场景通过。
- H5 版本、测试环境 URL、App 测试包版本已记录。
- 发布、灰度、回滚或暂不发布的原因已记录。

## 工作流程

### 1. 初始化 Base

当用户提供飞书多维表格地址后，AI 应：

1. 读取本工作流。
2. 读取 `.ai-workspace/templates/FEISHU_BASE_H5_DEMAND_SCHEDULE_SCHEMA.md`。
3. 创建或检查 7 张表。
4. 创建字段、单选项、关联字段、推荐视图、甘特图、项目协作总览和仪表盘。
5. 将 `.ai-workspace/plans/h5-demand-schedule-seed-v0.md` 中的首批页面和子需求写入 Base。

### 2. 新页面进入排期

每新增一个页面，必须新增一条 `H5 页面需求`，并同时判断：

- 是否需要后端接口。
- 是否需要原生 App 对接。
- 是否需要管理后台配置。
- 是否需要测试验收场景。

如需要，就在对应子表中创建记录并关联页面。

### 3. 团队补充信息

后端、原生、测试可以分别补充自己负责的表：

- 后端补接口路径、字段、错误码、测试数据、完成时间。
- 原生补 Bridge 方法、WebView 行为、iOS/Android 状态、测试包版本。
- 测试补用例、前置条件、环境、执行结果、缺陷链接。

### 4. 每周排期更新

每周至少更新一次：

- 页面状态。
- 子需求状态。
- 计划开始和计划完成。
- 阻塞项。
- 本周完成结果。
- 下周计划。

### 5. 归档和回写仓库

页面验收完成后：

- Base 中页面状态改为 `已完成`。
- 对应工作项、对接说明、契约和测试报告回写仓库。
- 如形成长期规则，补充 `.ai-workspace/` 或子项目 `docs/`。

## AI 使用要求

AI 操作飞书 Base 前必须确认：

- 用户提供的是正确的 Base URL。
- 本次是创建新 Base 结构，还是更新已有表结构。
- 是否允许新增表、字段、视图和记录。
- 是否需要保留已有人工字段和记录。

禁止事项：

- 禁止在未确认 Base 地址时直接创建线上多维表格。
- 禁止覆盖团队手工补充字段。
- 禁止把未确认接口写成已确认。
- 禁止删除已有记录，除非用户明确要求。
- 禁止只建页面表，不建后端、原生和测试子表。
