# 统一工作项结构

## 目的

工作项是 AI 和人类协作的最小正式单位。它记录目标、边界、上下文、契约、验收、测试、发布和责任归属。

## 命名

工作项文件使用以下格式：

```text
TASK-YYYY-MMDD-NNN-短标题.md
```

示例：

```text
TASK-2026-0601-001-product-detail-api.md
```

## 必填字段

每个工作项必须包含：

- 标题
- 状态
- 目标
- 背景
- 涉及项目
- 范围
- 不包含
- 责任边界
- 契约影响
- 对接说明
- 对方责任
- Mock 和联调方式
- 验收标准
- 验证命令
- 发布影响
- 风险和阻塞
- 变更记录

## 任务粒度

一个工作项应该能在一次清晰的开发周期内完成。满足以下任一条件时，应拆分：

- 涉及三个以上独立业务域。
- 同时改变 API、UI、发布链路和外部运行环境能力。
- 验收标准超过十条。
- 无法在一份实现计划中清楚表达步骤。

## 状态字段

状态必须使用 `.ai-workspace/STATE_FLOW.md` 中定义的枚举。禁止写模糊状态，例如“差不多完成”“待看看”“基本好了”。

## 契约字段

如果任务涉及跨项目交互，必须写明：

- 提供方。
- 消费方。
- 契约文档路径。
- 是否向后兼容。
- 是否需要迁移。
- 是否需要灰度。

## 对接字段

如果 H5 需求涉及 Java/API、外部运行环境、Java 配置平台或 CI/发布，必须写明：

- 对接说明路径。
- 需要确认的对方角色。
- 对方需要完成的事项。
- H5 侧 mock 或 fallback。
- 联调环境和联调步骤。
- 当前确认状态。

禁止只写“Java 提供接口”“外部环境提供能力”这类不可验收描述。

## 验收字段

验收标准必须可验证。禁止写：

- “体验更好”。
- “逻辑完善”。
- “页面正常”。

推荐写：

- “接口 404 时页面展示未找到状态”。
- “`pnpm test` 和 `pnpm typecheck` 通过”。
- “active manifest 仍直接返回 manifest body，不添加 wrapper”。
