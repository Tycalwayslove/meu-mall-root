# MeuMall AI 交付体系设计

## 背景

当前 MeuMall 工作区包含 H5、原生壳、manifest 服务、管理台和本地 CI。项目边界已经初步形成，但正式业务开发还缺少统一的 AI 协作机制。主要风险是新会话丢失上下文、跨项目说法不一致、接口契约滞后、验收标准不统一，以及发布责任边界不清。

## 设计目标

建立一套仓库优先的 AI 开发机制，让任何新会话都能通过仓库文档恢复上下文，并按统一工作项推进需求、接口、实现、测试、发布和归档。

## 核心方案

采用“根级统一规则 + 子项目本地事实”的结构：

- 根级 `.ai-workspace/` 负责统一工作模型、项目地图、状态流转、验收标准、契约治理、发布治理和术语。
- 子项目 `.ai/` 继续记录本项目状态、上下文、待办、测试报告和决策。
- 工作项作为正式开发入口，记录目标、范围、责任边界、契约影响、验收、验证和发布影响。

## 关键规则

- 没有工作项，不进入正式业务开发。
- 没有 `ready` 状态，不开始实现。
- 跨项目变更先写契约，再写实现。
- 没有验证记录，不声称完成。
- 有发布影响的任务必须说明灰度、回滚和 smoke check。

## 初始落地范围

第一阶段只建立机制文档，不修改业务代码：

- 创建 `.ai-workspace/` 根级规范。
- 创建工作项模板。
- 创建契约和领域模型目录入口。
- 更新各子项目 `AGENTS.md`，让它们先读取根级规范。

## 后续阶段

1. 补齐 manifest、release API、Native Bridge、H5-server API 四类根级契约。
2. 补齐 product、sku、cart、order、user 领域模型。
3. 选择一个小业务任务试运行完整流程，例如商品详情接入真实 API。
4. 根据试运行结果收紧模板、状态流转和验证规则。
